# دليل سريع للمشتركين الجدد في MedTracker

## 1) نظرة عامة
- المشروع عبارة عن **بوت تليجرام + لوحة تحكم محلية** لإدارة صلاحيات الأدوية.
- التشغيل محلي بالكامل مع **SQLite** (`pharmacy.db`) بدون خادم خارجي.
- أول تشغيل يمر عبر **Setup Wizard** لحفظ الإعدادات في `config.json`.

## 2) الهيكل العام
- `index.js`: نقطة الدخول، تحميل الإعدادات، تشغيل البوت، تفعيل الجدولة ولوحة التحكم.
- `src/setup.js`: معالج الإعداد الأول وحفظ/قراءة `config.json`.
- `src/db.js`: تهيئة قاعدة البيانات وإنشاء جدول `nupco_inventory` ودالة `sql` (tagged template).
- `src/commands/*`: أوامر تليجرام (`add`, `list`, `check`, `scan`).
- `src/scheduler.js`: التقرير اليومي التلقائي.
- `src/status.js`: لوحة تحكم `localhost:3000` (إحصائيات + إجراءات تشغيل).
- `src/utils/*`: وظائف مساعدة (تصنيف الأدوية، إدارة جلسات المسح).
- `.github/workflows/build.yml`: بناء نسخة Windows ونشرها في Releases عند tag.

## 3) مسار التدفق التشغيلي
1. التطبيق يبدأ من `index.js` ويقرأ الإعدادات من `src/setup.js`.
2. يتم تهيئة قاعدة البيانات عبر `src/db.js`.
3. يشتغل Telegram bot وتُربط أوامر `src/commands`.
4. يتم تشغيل:
   - الجدولة اليومية (`src/scheduler.js`)
   - لوحة التحكم المحلية (`src/status.js`)

## 4) أهم الأمور التي يجب معرفتها
- **مصدر الإعدادات الحقيقي** هو `config.json` (وليس `.env` في التشغيل العادي).
- أوامر المسح تعتمد على **Groq Vision/Text** واستخراج JSON ثم تأكيد المستخدم قبل الحفظ.
- يوجد كود API في `src/api/*` يبدو موجّهًا لواجهة ويب متعددة المستخدمين؛ راجع قبل استخدامه لأن بقية المشروع الحالية تتمحور حول تشغيل محلي بسيط.
- الجدولة تعمل بتوقيت UTC لضبط إرسال تقرير 9 صباحًا بتوقيت الرياض.
- المشروع يستهدف Windows في البناء النهائي عبر `pkg`.

## 5) ماذا تتعلم بعد ذلك؟ (خارطة مقترحة)
1. **ابدأ من مسار الاستخدام الكامل**:
   - `index.js` → `setup.js` → `commands/scan.js` → `db.js`.
2. **ثم استوعب الاستقرار التشغيلي**:
   - `status.js`, `scheduler.js`, `autostart.js`.
3. **ثم تحسينات الجودة**:
   - إضافة اختبارات فعلية (unit/integration) بدل `test.js` اليدوي.
   - توحيد نمط البيانات (مثل الحقول المستخدمة في API مقابل schema).
4. **ثم تحسينات المنتج**:
   - تسجيل أخطاء موحد.
   - فصل منطق Groq في خدمة مستقلة.
   - وثائق مساهمة (`CONTRIBUTING.md`) مع أسلوب الكود وسير العمل.
